# AutoDaily Frontend Design Guide

本文档整理当前前端重写过程中已经明确的视觉风格、交互规则、页面结构原则与工程注意事项。后续页面设计、组件拆分和样式调整应以本文档为准。

## 1. 产品定位

AutoDaily 是桌面端自动化工具，不是营销官网，也不是通用 SaaS 后台。

技术形态：

- Vue 3 + TypeScript + Vite
- Tauri 桌面壳
- 专业工具型工作台 / 控制台 / 配置工具

核心任务：

- 管理多个设备
- 给设备分配脚本任务
- 启动 / 暂停 / 停止任务
- 管理本地脚本与云端脚本
- 查看运行日志
- 配置系统设置
- 提供脚本编辑器入口

## 2. 总体设计目标

设计语言参考 Apple 官网、iOS 18+、最新 macOS 的秩序感与材质感，但必须服务于专业桌面工具的效率。

目标关键词：

- 内容优先
- 克制、安静、整洁
- 高级但不过度展示
- 有系统感，不像网页后台
- 桌面端长时间使用舒适
- 视觉精致，但不牺牲操作效率

一句话定义：

具有 Apple 质感的专业自动化控制台。

## 3. 明确禁止项

以下方向禁止继续出现：

- 通用 SaaS 后台模板感
- 大面积紫色渐变主视觉
- 大量默认卡片堆叠
- 炫技 3D 或霓虹灯效果
- 金融仪表盘风格
- 电商平台风格
- 营销官网 Hero 风格
- 为了模仿 Apple 而牺牲信息效率

## 4. 视觉语言

### 4.1 色彩

- 主色克制，偏系统感，以中性灰、冷白、轻蓝灰为主
- 强调色只用于关键激活状态和主操作，不要泛滥
- 成功、警告、错误状态要清楚，但不能刺眼
- 页面背景与内容层必须拉开对比，不能出现容器边界不清的情况
- 列表选中态必须明显，且 `selected` 与 `selected:hover` 不能退化成普通 `hover`

当前具体要求：

- 选中列表项、侧边栏项、下拉项的 `default / hover / active / active:hover` 必须保持同一套状态逻辑
- 同状态下背景色和鼠标样式要一致
- 优先参考当前 `list-item` 的状态颜色体系

### 4.2 圆角

- 参考 macOS / iOS 的柔和圆角
- 按钮、列表项、输入框、面板使用可辨识但不过分夸张的圆角
- 禁止继续使用胶囊感过强的大圆角按钮

### 4.3 边界、阴影与分层

- 用轻边框、半透明分割线、柔和阴影建立层次
- 不依赖厚重描边和重卡片
- 页面背景和内容层要有明确层级差
- 面板不是传统后台的硬卡片，而是“轻容器 + 材质分层”

### 4.4 材质

- 毛玻璃只能用于关键层：侧边栏、工具栏、下拉、浮层、模态框
- 下拉弹层应带模糊与半透明材质，不能是生硬的纯色块
- 弹层必须脱离父容器裁剪和遮挡，不能被其他区域盖住

## 5. 排版与信息密度

- 保持桌面端合适的信息密度，不要空，也不要挤
- 通过留白、字重、透明度和分组来建立层级，不靠大面积边框
- 大块区域优先做结构分区，不要把所有内容堆成一列
- 面对长列表或复杂详情时，优先考虑分栏、切换、折叠、tab

## 6. 动效与反馈

动效原则：

- 快
- 轻
- 系统化
- 不喧宾夺主

具体要求：

- 页面切换用轻量过渡，不用营销式动画
- 列表 hover、active、selected 反馈清晰
- 选中态在 hover 后不能丢失
- 弹窗出现与关闭要平滑，但节奏短
- 状态切换要有细微反馈，不要做夸张缩放

## 7. 页面级设计原则

### 7.1 主框架 / 导航

- 整体框架要像原生桌面应用，不像网页后台
- 左侧导航带轻微毛玻璃和半透明层级
- 顶部区域承担页面标题、状态与全局入口
- 用户信息、版本信息放在侧边或底部辅助区域

### 7.2 任务管理

这是主工作台，必须体现“控制中心”气质。

要求：

- 一眼看出整体运行状态、在线设备、空闲设备
- 不适合把所有设备长列堆在同一列
- 优先使用“设备切换 + 当前设备详情”或“可折叠设备区”结构
- 当设备和任务较多时，后面的设备不能难以查看

当前实现方向：

- 左侧设备切换
- 右侧当前设备详情

### 7.3 设备列表

- 结构应接近 macOS 设置页 + 数据列表
- 浏览要清晰，新增 / 编辑弹窗要有系统设置感
- 表单分组要稳定，不要杂乱堆字段

### 7.4 本地脚本列表

这是资源管理器 / 内容浏览器 / 配置面板的组合界面。

要求：

- 不应只是“左侧列表 + 右侧脚本信息”
- 必须能同时表达三者关系：脚本列表、脚本详情、任务设置
- 优先三栏布局
- 中栏操作区不要被一排文字按钮挤满，动作应尽量紧凑，优先图标化

当前结构方向：

- 左侧：脚本列表
- 中间：脚本详情
- 右侧：任务设置 / 变量配置

### 7.5 ScriptInfoDialog

脚本信息内容较多，禁止继续单列堆叠。

要求：

- 使用左侧 tab 分组
- 至少分成：基本信息、模型信息、赞助信息
- 每个 tab 内再按逻辑分区布局

### 7.6 运行日志

- 参考开发工具中的日志面板
- 长时间阅读低干扰
- 筛选控件、下拉样式与全局风格一致
- 状态色轻量，不花哨

## 8. 组件与样式约束

### 8.1 组件拆分

- 单个组件不要堆太多内容
- 写前先判断是否可复用
- 写后再检查是否值得抽离
- 多处复用的内容放入共享组件目录

### 8.2 目录规则

- `views` 按页面组织
- 同一级页面放在 `views` 下同级目录
- 页面内部子内容放在该页面同目录子文件夹
- 跨页面复用组件放共享组件目录

示例：

- 脚本列表、任务管理、脚本编辑是同级页面
- 策略编辑等属于脚本编辑内部内容，应放到脚本编辑目录内部

### 8.3 Select / 列表 / 侧边栏统一规范

- 自定义下拉、列表项、侧边栏项的状态体系要统一
- 对应状态的背景色、文字色、鼠标样式保持一致
- 弹层宽度跟随触发器，不允许铺满整个窗口
- 弹层必须高于其他内容层

## 9. 数据与前后端约束

### 9.1 数据来源

- 数据的来源、共享和持久化要统一设计
- 不能因前端暂时缺字段就直接写 `any`
- 前端缺少信息时，要优先看后端 `lib.rs` 命令接口和服务端 `adbackend`

参考优先级：

1. 当前 Tauri Rust 命令
2. `adbackend` 服务端实现
3. `src_bak_sec` 历史前端，仅作参考，不可照搬

### 9.2 状态管理

- 持久化数据用 `@tauri-apps/plugin-store`
- 共享内存状态用 Pinia
- 涉及服务端交互、认证、上传、敏感流程，尽量下沉到 Tauri Rust 层

### 9.3 类型约束

- 优先使用 `types/bindings` 导出的真实类型
- 如果绑定类型存在前端不可直接使用的问题，再做最小必要的前端适配层
- 不允许以 `any` 作为结束方案

## 10. 交互实现注意事项

- 不清楚的地方先沟通，或先查历史代码
- 复杂业务不要只看前端，应确认服务端真实逻辑
- 上传、认证、资料回填、token 刷新等与服务端密切相关的流程，优先放 Rust 后端命令层处理
- 页面实现既要符合视觉方向，也要经得起真实业务推演

## 11. 开发过程注意事项

- 扫描代码时默认排除 `node_modules`、`target`、`dist`
- 只在必要时查看历史目录
- 样式调整优先统一到共享组件和全局规则，避免局部临时补丁越积越多

## 12. 当前已确认的实现倾向

以下方向后续不要反复摇摆：

- 任务管理不用超长单列设备堆叠
- 本地脚本页保持三栏关系
- ScriptInfoDialog 使用左侧 tab
- 按钮圆角收紧
- 下拉弹层带模糊材质
- 列表选中态明显且稳定
- 页面背景与内容层拉开
- 与服务端交互的关键流程尽量后移到 Rust 侧
