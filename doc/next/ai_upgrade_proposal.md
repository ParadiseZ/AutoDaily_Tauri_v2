# AutoDaily 二期升级方案 — AI 驱动的自适应游戏自动化

> 编写日期: 2026-03-05
> 核心目标: 从"脚本驱动（规则写死）" → "AI 驱动（自适应理解+决策）"

---

## 一、核心理念

### 当前一期架构

```
开发者编写脚本 → 节点编排策略 → 固定流程执行 → 截图+OCR/YOLO匹配
```

**问题**: 每个游戏、每个任务都需要开发者手动编排节点。游戏更新后可能失效。

### 二期目标架构

```
用户提供包名 → AI 观察屏幕 → 理解当前状态 → 规划下一步行动 → 执行 → 验证结果 → 循环
```

**关键转变**: 大模型成为"大脑"，现有 ADB/截图/OCR/YOLO 成为"眼睛和手"。

---

## 二、不需要微调

### 为什么不需要

1. **现代多模态模型**（GPT-4o、Gemini 2.5、Claude 3.5 等）已经具备：
   - 理解游戏截图的能力（识别按钮、文字、UI 元素）
   - 理解游戏任务逻辑的能力（"每日签到"、"限时活动"等概念通用）
   - 输出结构化动作指令的能力

2. **微调的代价**高、且游戏种类太多 — 不具备可扩展性

3. **更好的方案**: 提示工程（Prompt Engineering）+ 上下文注入 + 工具调用（Function Calling）

### 实际做法

| 方法                 | 说明                                                     |
| -------------------- | -------------------------------------------------------- |
| **System Prompt**    | 定义 AI 的角色、能力边界和输出格式                       |
| **游戏知识库**       | 每个游戏一份"知识文档"，描述 UI 结构、任务类型、关键界面 |
| **截图分析**         | 将当前截图发给多模态模型，让它描述看到的内容             |
| **Function Calling** | 定义可用动作（点击、滑动、返回、等待），AI 选择调用      |
| **历史上下文**       | 保持最近 N 步的操作记录，让 AI 了解当前进度              |

## 三、系统架构设计

### 整体流程

```
┌─────────────────────────────────────────────────────┐
│                    用户设置层                          │
│  包名 → 游戏Profile → 时间模板 → 任务偏好             │
└────────────────────┬────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────┐
│                  AI 决策层（核心）                      │
│                                                      │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │ 观察模块  │→│ 规划模块  │→│ 执行模块  │           │
│  │ (Vision) │  │(Planner) │  │(Executor)│           │
│  └──────────┘  └──────────┘  └──────────┘           │
│       ↑              ↑              │                │
│       │         ┌────┴────┐        │                │
│       │         │ 游戏知识库│        │                │
│       │         └─────────┘        │                │
└───────┼────────────────────────────┼────────────────┘
        │                            │
┌───────┼────────────────────────────┼────────────────┐
│       │        设备交互层           ▼                 │
│  截图/OCR/YOLO ←───────── ADB点击/滑动/返回          │
│  （现有一期能力）                                     │
└─────────────────────────────────────────────────────┘
```

### 决策循环（核心 Loop）

```
while 未完成所有任务:
    1. 截图
    2. OCR + YOLO → 结构化场景描述
    3. 组装 Prompt（系统提示 + 游戏知识 + 历史操作 + 当前截图 + 用户偏好）
    4. 调用多模态 LLM → 返回结构化动作
    5. 执行动作（点击/滑动/等待）
    6. 等待画面稳定
    7. 记录操作日志 + 更新任务完成状态
    8. 判断是否需要继续
```

## 四、需要向 AI 提供的数据

### 4.1 每次决策调用时提供

| 数据               | 格式                                                  | 说明                            |
| ------------------ | ----------------------------------------------------- | ------------------------------- |
| **当前截图**       | Base64 图片                                           | 多模态模型直接分析              |
| **OCR 结果**       | JSON 数组 `[{text, x, y, w, h}]`                      | 屏幕上的所有文字和位置          |
| **YOLO 检测**      | JSON 数组 `[{label, x, y, w, h, conf}]`               | 检测到的 UI 元素                |
| **操作历史**       | 最近 10-20 步 `[{action, target, result, timestamp}]` | 防止 AI 重复操作                |
| **当前任务上下文** | 文本                                                  | "正在执行签到任务，已完成第2步" |
| **用户偏好**       | JSON                                                  | 时间模板、优先级、跳过项        |

### 4.2 游戏知识库（每个游戏一份，一次性/低频更新）

```json
{
  "package_name": "com.example.game",
  "game_name": "某某手游",
  "knowledge": {
    "main_screens": [
      {
        "name": "主界面",
        "description": "游戏主界面，底部有导航栏",
        "key_elements": ["冒险", "背包", "商店", "活动", "邮件"],
        "entry_condition": "看到角色站在主城"
      }
    ],
    "daily_tasks": [
      {
        "name": "每日签到",
        "path": "主界面 → 活动 → 签到",
        "completion_indicator": "签到按钮变灰 或 显示'已签到'",
        "estimated_steps": 3
      },
      {
        "name": "体力收取",
        "path": "主界面 → 邮件 → 领取",
        "completion_indicator": "无可领取邮件"
      }
    ],
    "event_tasks": [
      {
        "name": "限时活动",
        "detection": "主界面右侧浮动图标 或 活动页红点",
        "type": "dynamic",
        "note": "活动内容随版本变化，需AI自行判断"
      }
    ],
    "navigation_rules": ["返回主界面：连续点击左上角返回按钮 或 点返回键", "关闭弹窗：点击右上角X 或 空白区域"]
  }
}
```

> **重要**: 这份知识库不需要覆盖所有场景。AI 看到截图后可以自行推理未知界面。知识库的作用是提供"快速路径"，减少探索和 API 调用次数。

### 4.3 游戏知识库的获取方式

| 方式            | 说明                                                    |
| --------------- | ------------------------------------------------------- |
| **AI 自动生成** | 首次运行游戏时，AI 浏览各个界面，自动记录导航路径和任务 |
| **社区共享**    | 用户可以导出/上传知识库到脚本市场，类似现有的脚本共享   |
| **手动编写**    | 开发者用户可以手动编写或修正知识库                      |
| **持续学习**    | AI 每次执行时发现新界面/新任务，自动更新知识库          |

## 六、技术实现方案

### 6.1 LLM 调用层

```rust
// 新模块: src-tauri/src/infrastructure/ai/
pub mod ai_client;       // LLM API 调用（支持多家厂商）
pub mod prompt_builder;   // Prompt 组装器
pub mod action_parser;    // 解析 AI 返回的动作指令
pub mod game_knowledge;   // 游戏知识库管理
pub mod task_tracker;     // 任务完成追踪
pub mod ai_scheduler;     // AI 驱动的调度器（替代/并行于现有 scheduler）
```

### 6.2 AI 输出的动作格式（Function Calling）

```json
{
  "thought": "当前在主界面，看到活动按钮有红点，应该先进入活动页面查看",
  "action": {
    "type": "click",
    "target": "活动按钮",
    "coordinates": [850, 420],
    "confidence": 0.95
  },
  "task_state": {
    "current_task": "检查活动任务",
    "progress": "navigating",
    "tasks_completed": ["签到"]
  }
}
```

### 6.3 与现有架构的兼容

| 现有组件        | 二期角色                            |
| --------------- | ----------------------------------- |
| ScriptExecutor  | 保留，用于"脚本模式"（兼容一期）    |
| ScriptScheduler | 保留，新增 `AiScheduler` 并行       |
| OCR/YOLO        | 作为 AI 的"传感器"，提供结构化输入  |
| ADB 操作        | 作为 AI 的"执行器"，执行点击/滑动等 |
| IPC 通信        | 不变，AI 调度运行在子进程中         |
| 前端            | 新增 AI 模式设置页、任务完成看板    |

### 6.4 运行模式切换

```
设备设置中新增: 运行模式
  ├── 脚本模式（一期）— 使用现有的脚本编排执行
  └── AI 模式（二期）— 使用大模型观察+决策
```

## 七、成本与性能考量

### API 调用成本估算

| 场景          | 每次调用Token | 调用次数/任务 | 成本估算（GPT-4o价格） |
| ------------- | ------------- | ------------- | ---------------------- |
| 截图分析+决策 | ~1000 tokens  | 10-30次/游戏  | ¥0.1-0.3/游戏/天       |
| 多设备×多游戏 | -             | -             | 10设备×3游戏 ≈ ¥3-9/天 |

### 性能优化策略

| 策略             | 说明                                                   |
| ---------------- | ------------------------------------------------------ |
| **分层决策**     | 简单状态（如弹窗关闭）用本地规则，复杂状态才调用 LLM   |
| **缓存相似场景** | 相同截图特征 → 缓存决策结果，减少 API 调用             |
| **批量规划**     | 一次性规划多步操作，而非每步都调用 API                 |
| **本地小模型**   | 用 ONNX 部署轻量模型做初步分类，只在不确定时调用大模型 |
| **OCR+YOLO优先** | 能用传统视觉解决的（如文字匹配）不调用 LLM             |

### 推荐的分层策略

```
Level 0: 纯规则匹配（弹窗关闭、已知按钮点击） → 0 cost
Level 1: 本地 ONNX 小模型（界面分类、状态判断） → 本地运行
Level 2: OCR+YOLO → 结构化场景 → 关键词匹配预设动作 → 低 cost
Level 3: 多模态 LLM API（未知场景、复杂决策） → 按需调用
```

## 九、风险与注意事项

| 风险                | 应对                                                     |
| ------------------- | -------------------------------------------------------- |
| AI 幻觉（点错位置） | 执行前二次确认：AI点击坐标 vs YOLO检测框，偏差过大则跳过 |
| API 费用超标        | 设置每日/每月费用上限，超限自动切换为脚本模式            |
| 游戏反作弊检测      | 保持现有随机延迟和人类化操作模式                         |
| API 延迟导致执行慢  | 缓存+批量规划+本地模型分层                               |
| 不同大模型能力差异  | 抽象统一接口，支持多厂商切换                             |

## 十、总结

**核心结论**: 不需要微调模型。通过"多模态 LLM API + 结构化 Prompt + 游戏知识库 + 分层决策"的组合，可以实现你描述的自适应游戏自动化。

**最大优势**: AI 模式可以处理**未知的新活动和界面变化**，这是脚本模式无法做到的。

**与一期的关系**: 二期是在一期基础上的**增量升级**，不是推翻重做。现有的截图/OCR/YOLO/ADB 基础设施完全复用，只是在"决策层"引入了 AI。
