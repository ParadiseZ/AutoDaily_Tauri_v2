# AutoDaily Frontend Engineering Guidelines

本文档整理前端实现过程中的组件拆分、目录组织、状态管理、类型约束和前后端分工规则。后续重写时应以本文档为准。

## 1. 组件与目录规则

### 1.1 组件拆分

- 单个组件不要堆太多内容
- 写前先判断是否可复用
- 写后再检查是否值得抽离
- 多处复用的内容放入共享组件目录

### 1.2 目录规则

- `views` 按页面组织
- 同一级页面放在 `views` 下同级目录
- 页面内部子内容放在该页面同目录子文件夹
- 跨页面复用组件放共享组件目录

示例：

- 脚本列表、任务管理、脚本编辑是同级页面
- 策略编辑等属于脚本编辑内部内容，应放到脚本编辑目录内部

### 1.3 Select / 列表 / 侧边栏统一规范

- 自定义下拉、列表项、侧边栏项的状态体系要统一
- 对应状态的背景色、文字色、鼠标样式保持一致
- 弹层宽度跟随触发器，不允许铺满整个窗口
- 弹层必须高于其他内容层

## 2. 数据与前后端约束

### 2.1 数据来源

- 数据的来源、共享和持久化要统一设计
- 不能因前端暂时缺字段就直接写 `any`
- 前端缺少信息时，要优先看后端 `lib.rs` 命令接口和服务端 `adbackend`

参考优先级：

1. 当前 Tauri Rust 命令
2. `adbackend` 服务端实现
3. `src_bak_sec` 历史前端，仅作参考，不可照搬

### 2.2 状态管理

- 持久化数据用 `@tauri-apps/plugin-store`
- 共享内存状态用 Pinia
- 涉及服务端交互、认证、上传、敏感流程，尽量下沉到 Tauri Rust 层

### 2.3 类型约束

- 优先使用 `types/bindings` 导出的真实类型
- 如果绑定类型存在前端不可直接使用的问题，再做最小必要的前端适配层
- 不允许以 `any` 作为结束方案

## 3. 交互实现注意事项

- 不清楚的地方先沟通，或先查历史代码
- 复杂业务不要只看前端，应确认服务端真实逻辑
- 上传、认证、资料回填、token 刷新等与服务端密切相关的流程，优先放 Rust 后端命令层处理
- 页面实现既要符合视觉方向，也要经得起真实业务推演

## 4. 开发过程注意事项

- 扫描代码时默认排除 `node_modules`、`target`、`dist`
- 只在必要时查看历史目录
- 样式调整优先统一到共享组件和全局规则，避免局部临时补丁越积越多

## 5. 当前已确认的实现倾向

- 与服务端交互的关键流程尽量后移到 Rust 侧
- 本地脚本上传前的作者校验、认证校验、资料回填等逻辑优先放 Rust 命令层
- 前端只保留 UI 态、页面编排和必要的内存状态，不要把后端敏感流程摊在页面组件里

## 6. 脚本编辑器工作区规则

### 6.1 基础布局

- 脚本编辑器主体保持三栏布局
- 左侧用于任务列表或后续的策略类导航
- 中间用于当前对象的配置和步骤模板源
- 右侧用于当前层级的工作区，不把新增入口拆成独立弹层流程
- 页面高度以窗口可视区为准，左、中、右三栏各自滚动，不共用整页滚动条

### 6.2 中间区域规则

- `基本 / 输入 / 界面 / 步骤` 使用单行 tab 样式，不使用说明卡片或大段描述文案
- 中间区域的 `步骤` tab 是步骤模板的唯一插入源
- 顶层步骤、`Then`、`Else`、`While`、`For` 等所有层级统一从中间模板区插入
- 右侧工作区不再额外放独立的“添加步骤”按钮
- 复制不是第一阶段核心能力，优先保证插入、删除、排序、编辑链路完整

### 6.3 右侧步骤工作区规则

- 右侧工作区分为两部分
- 左半区显示当前层级的步骤列表
- 右半区显示当前选中步骤的详情编辑
- 条件分支、循环、顺序步骤等节点不是只显示摘要，而是可以进入对应层级继续编辑
- 步骤列表样式贴近现有 list-item 体系，保证选中、悬停、拖动态统一

### 6.4 面包屑与层级导航

- 右侧顶部必须提供可点击面包屑
- 面包屑用于在顶层、分支层、当前节点之间跳转
- 用户点击某个面包屑节点后，应直接切回该层级的步骤列表
- 面包屑不仅是展示路径，还承担层级导航职责

### 6.5 嵌套步骤的前端状态模型

- 不改后端 `Step` 结构体和持久化数据结构
- 前端通过路径模型表达当前编辑位置，而不是只保存顶层索引
- 至少区分两个状态：当前激活分支路径、当前选中步骤路径
- 对步骤的读取、插入、删除、排序、更新，都通过路径工具统一完成
- `Then`、`Else`、`While`、`For`、顺序步骤、命中后行为等分支，统一走同一套路径编辑模型

### 6.6 条件与循环节点规则

- `if / while / for` 的条件区和步骤区分开处理
- 条件区负责条件类型、比较逻辑和表达式编辑
- 分支区或循环体负责嵌套步骤列表及其详情编辑
- 用户进入 `Then`、`Else` 或循环体后，中间模板区插入的步骤应落到当前激活层级

### 6.7 记录目标

- 脚本编辑器应从“顶层步骤编辑”升级为“路径驱动的层级编辑”
- 用户在任意层级都能完成同一套核心动作：进入层级、插入步骤、删除步骤、拖动排序、编辑详情
- 不允许出现“顶层能编辑，嵌套层只能查看摘要”的交互断层

### 6.8 步骤工作区组件化

- 步骤工作区不能继续堆在任务页面单组件里
- 任务页、策略页后续都要复用同一套步骤工作区
- 至少拆成：面包屑、步骤列表、步骤详情、搜索规则/条件等子编辑器
- 页面组件负责编排和状态注入，步骤工作区负责步骤相关交互，不把任务专属逻辑写死在步骤组件里
- 步骤详情优先按快速模板分类拆分组件，至少对应：`动作 / 流程 / 数据 / 状态 / 视觉 / 容器`
- `EditorStepWorkspace` 保留路径、选中态、分支导航和更新分发，不继续堆具体分类表单
- 步骤相关组件和工具统一收在 `src/views/script-editor/editor-step/`，不要再平铺回 `script-editor` 根目录

### 6.9 步骤常量与类型来源

- `Step.op`、`Action.ac`、`FlowControl.type`、`DataHanding.type`、`TaskControl.type`、`VisionNode.type` 等取值必须统一收口
- 不允许在多个组件里直接散写同一批字符串常量
- 优先从单一常量定义文件统一导出，再在模板、描述函数、路径工具、详情表单中复用
- 搜索规则、状态目标、状态类型、过滤模式等衍生枚举也要走同一套常量源

### 6.10 步骤列表与面包屑表现

- 步骤列表的选中态必须明显，当前激活步骤要有 active 样式
- 面包屑独占一行，不和步骤列表混排
- 面包屑要有真正的面包屑视觉，不使用普通按钮样式
- 面包屑的最后一项应反映当前激活步骤标题或名称
- 用户从嵌套层点击面包屑父级返回时，对应步骤应保持 active
- 拖动排序必须是实际可用的交互，不能出现鼠标拖动无效或禁止态

### 6.11 叶子步骤表单覆盖目标

- 右侧步骤详情要尽量覆盖常用叶子结构，不要让用户频繁退回 JSON
- 第一优先级包括：`sequence.reverse`、`continue / break`、`taskControl`、`visionSearch.rule`、`setVar / getVar` 的补充字段、`filter` 的主要字段
- `setVar / getVar / varCompare` 里的值编辑必须显式选择 `Int / Float / Bool / String`
- `if / while / for` 里的 `varCompare.var_name` 不能手填，必须从变量目录里选择
- 不允许只靠当前值内容反推类型，例如 `0` 不能自动把用户刚选的 `Float` 又退回成 `Int`
- 条件规则、搜索规则这类递归结构，要抽成独立子编辑器，不要硬塞回主详情组件
- 新增叶子表单后，必须补对应的保存回归测试

### 6.12 特殊分支的一致化规则

- 只要语义上属于“可进入的子步骤层级”，UI 就应尽量和 `Then / Else / 循环体` 保持一致
- `filter.then_steps` 按多步骤分支处理，和 `if / while / for / visionSearch` 的嵌套步骤工作区一致
- `filter.then_steps` 默认从空列表起步，不预塞占位步骤
- 用户进入 `filter.then_steps` 后，仍然通过中间模板区插入、删除、排序和编辑当前层级的步骤
- 即使分支名称或业务含义不同，也不应额外做一套特殊表单入口
- 前端路径系统可以保留分支种类差异，但交互形式保持统一

### 6.13 变量目录规则

- 脚本编辑器需要统一的变量目录，不能继续依赖用户手填变量名字符串
- 变量目录的存储位置优先放在脚本数据 JSON 中，作为脚本元数据的一部分
- 变量目录的类型来源以 Rust 结构体为准，再通过 `ts-rs` 导出到前端
- 变量目录至少区分 `input`、`runtime`、`system` 三类命名空间
- `input` 表示用户可配置、可持久化、可绑定 UI 的输入变量
- `runtime` 表示步骤产物和执行过程中的中间变量
- `system` 表示引擎注入的只读上下文变量
- 变量目录定义和变量值要区分，当前优先管理“变量定义”，不单独设计运行时值表
- UI 字段绑定、`setVar`、`varCompare`、后续表达式补全，都应从同一份变量目录取数
- 变量目录字段要包含稳定 `id`，不要只靠 `key` 文本串联绑定关系
- Rust 结构体字段必须带注释，说明每个字段的职责，方便 TS 导出后同步保留语义
- 任务输入的“当前值”与脚本级变量目录的“变量定义”要分层保存
- `task.data.variables` 负责保存当前任务的输入默认值
- `script.data.variableCatalog` 负责保存输入变量定义、命名空间、类型、说明和绑定能力
- 当前任务编辑输入时，要同时同步这两层数据，不允许只改其中一处
- 变量目录里的备注是给开发期索引、绑定、后续引用看的，不是写给脚本发布页或终端用户的说明文案
- 当前阶段不臆造大批 `system.*` 内建变量；只有后端真实稳定注入到 `scope` 的系统变量，才允许进入目录和建议列表
- `runtime.*` 变量建议可以从步骤输出自动推导，例如截图输出、视觉搜索输出、过滤结果输出
- 输入工作区按“两栏”组织：中间是变量列表，右侧是当前选中变量的详情
- 点击“添加变量”后，中间列表新增变量元素；点击列表项后，右侧才显示该变量的名称、键、类型、备注、作用域等字段
- 作用域属于变量定义本身，至少区分 `input / runtime / system`
- 键和作用域分离编辑，前端详情里只编辑局部 key；持久化到目录时再拼成完整变量名，例如 `input.xxx / runtime.xxx / system.xxx`
- 只有 `input` 作用域在右侧详情中显示“默认值”编辑；`runtime / system` 只维护定义和元数据

### 6.14 setVar / getVar 规则

- `setVar` 不是自由拼装表单，优先收敛成三种写入方式：`固定值`、`引用变量`、`表达式`
- 表达式能力保留，但不应与其他写入方式同时平铺显示
- `setVar / getVar / varCompare` 的值类型编辑必须显式选择 `Int / Float / Bool / String`
- 变量选择应接入变量目录，并根据 `readable / writable / valueType / namespace` 做过滤
- `setVar` 的目标变量优先从变量列表对应的变量目录里选择，不再把“手填变量键”当主流程
- `varCompare` 的变量名也要走变量目录选择，不再保留自由输入框
- 当 `setVar` 的目标变量已在变量目录中登记时，界面优先收敛成“目标变量 + 值”
- 目标变量如果是 `int / float / bool / string` 这类标量类型，固定值编辑器直接跟随变量类型生成，不再重复让用户先选值类型
- 目标变量如果是复杂类型，或当前数据仍是兼容旧结构，则退回表达式输入或兼容输入框
- 当前运行时 `scope` 是全局可见的，`getVar` 在现有执行模型下不是高频主能力
- `getVar` 保留兼容，但应从主流程和高频模板中降级，不再占据核心编辑空间
- 步骤模板分组里应把 `getVar` 这类兼容或低频能力放到靠后的低频分组，避免和高频动作、流程、数据处理混在同一层级

### 6.15 UI 绑定规则

- UI 字段绑定关系优先使用变量目录里的稳定 `variableId`
- 为兼容旧数据，可以保留 `inputKey` 作为回填和迁移字段，但新逻辑不能只依赖 `inputKey`
- UI 预览和字段详情读取变量值时，应优先按 `variableId` 查找，再回退到 `inputKey`
- UI 绑定选项只展示 `uiBindable = true` 的变量，不把不适合界面绑定的复杂变量暴露给用户
- UI 字段详情里，绑定关系要直接展示变量的 `namespace / valueType / key / description`
- UI 预览只展示最终设置界面里真实会出现的控件，不把字段结构信息、绑定关系、内部键值混进预览
- `text` 控件在预览区按静态文本处理，适合做后缀说明或提示文本，不伪装成输入框
- “是否可编辑”只作用于 `text` 控件，用来决定文本变量在设置页中是输入框还是静态说明文本
- `checkbox` 控件应支持 `checkbox / switch` 两种显示样式，这只是同一字段的展示变体，不单独拆成新控件类型
- `select` 预览应展示当前值或首个可选项，视觉上接近真实下拉框，不展示编辑器内部字段信息
- `radio` 预览应展示互斥选项本身，并用明确的选中态区分当前值，不退化成普通标签云
- UI 字段详情不要一列到底，至少按“字段本体 / 变量绑定 / 展示内容”分组

### 6.16 visionSearch 规则编辑器

- `visionSearch` 的“搜索规则”区域必须把“逻辑组”作为真正的容器来编辑，不要把规则和组混成同一层视觉
- 可视化编辑时默认以“根逻辑组”开场，而不是让根节点直接变成任意规则类型
- 只有逻辑组才提供“添加规则 / 添加子组”入口
- 单个规则项只负责自身参数编辑，不承担组合职责
- 每个逻辑组都应有独立卡片、缩进或层级背景，保证规则增多时仍能分辨归属
- 逻辑组需要明确展示 `逻辑运算`、`搜索作用域`、收起展开和删除等控制项
- 搜索规则区域的视觉分组要明显区分：基础信息、搜索规则、命中后行为，避免所有字段挤成一整块

### 6.17 UI 与时间模板值

- 脚本编辑器里的 `uiData` 只负责描述变量怎么给用户配置，不负责存某个用户最终选了什么值
- UI 字段绑定的是变量定义，运行时真正读取的是变量，而不是 UI 字段本身
- 时间模板只表示普通用户定义的时间段，不包含脚本变量值
- “某个脚本在某个时间模板下的一整套变量值”必须单独存储，不要塞回 `uiData`
- 这层值的归属是 `script_id + time_template_id` 的组合，不是 `time_template` 单体，也不是 `script` 单体
- 变量值建议按 `variableCatalog` 里的稳定 `variableId` 持久化，不按 UI 字段 key 持久化
- `device_script_assignments.time_template_id` 只负责表明设备当前使用哪个时间模板，不承担模板变量值的存储职责
- `account_data` 继续保留作设备或账号扩展数据，本轮不占用
- 当前脚本编辑器只负责“变量定义层”和“UI 绑定层”，不负责模板值保存
- 当前 UI 预览的目标应更接近最终设置界面，而不是展示编辑器内部字段结构
