# 功能设计或优化提示词

更新时间：2026-06-08

本文不是某个单一项目的需求文档，而是一份可复用的工作约束。目标是减少“局部补丁式实现”，在任何项目里都先把全局边界、状态归属、真实流程和 UI 体感想清楚，再落代码。

## 一、适用场景

以下情况默认先按本文思考，再进入实现：

- 新功能设计
- 现有流程改造
- 状态流、调度流、连接流、运行流重构
- 管理后台、控制台、配置页、编辑器、运行面板这类长期使用界面的优化

## 二、必须先回答的问题

在开始写代码前，先在当前项目真实上下文里回答这些问题：

1. 这次新增的内容，应该挂在哪个已有 owner 下最方便维护？
2. 这个功能在当前项目的实际流程里，前一步和后一步分别是什么？
3. 它依赖哪些前提条件？
4. 它产生哪些状态？这些状态由谁持有、谁更新、谁消费？
5. 有没有形成第二套 source of truth？
6. 失败后，账本、流程、状态、UI 各自该怎么落结果？
7. 这段 UI 是给谁用的？在那个角色眼里，最重要的信息是什么？
8. 当前展示是否简洁、稳定、低认知负担，而不是“信息很多但不好用”？

## 三、架构视角约束

这是偏实现侧的硬约束：

- 先看项目现有 owner，不要先造新层。
- 一个运行事实只能有一个主 owner；不要拆成多套并行状态。
- 先定义 source of truth，再定义事件，再定义 UI 映射。
- 前后端对同一业务事实，应共享同一份合同模型，不要后端一套、前端再手写一套近似结构。
- 后端 domain / DTO / 事件 payload 一旦作为合同输出，前端应优先直接复用生成类型或同源类型，不要自己重新发明字段名、状态名、层级结构，除非有其它用途，比如枚举类，导出只有类型限制，但select数组需要所有类型。
- 先确认边界，再确认职责，再确认代码落点。
- 异步流程、后台任务、连接链路、执行链路要按单对象独立思考，不要混成一个全局大状态。
- 业务状态、连接状态、运行状态、内部控制状态要分层建模。
- 内部英文状态和 UI 展示文案要分层，不要互相污染。
- 状态字段、事件类型、命令参数，能用枚举、布尔、数字、结构化对象表达，就不要退化成字符串常量。
- 字符串只用于真正开放文本或天然字符串字段，例如名称、路径、消息、时间文本、原始外部协议值；不要把可枚举语义长期存成随手比较的裸字符串。
- 任何 watcher、cache、signal、临时 map，如果只是为了补 owner 不清的问题，应先停下来重构状态归属。

## 四、项目语境视角约束

这是偏产品和实际使用侧的约束，同样成立：

- 设计必须建立在当前项目的真实流程上，不做脱离场景的抽象，除非是新架构设计。
- 必须检查前提条件，例如：
  - 某个动作开始前，对应依赖是否已经就绪
  - 某个流程是否必须等待上一阶段成功
  - 自动流程与手动触发的语义是否被混用
- 状态呈现要能解释“现在为什么不能继续”，不能只刷新一下 UI。
- 高频操作页优先保证直观、稳定、可扫读。
- 同一块信息不要在多个地方用不同术语表达。
- 用户最关心的是“当前对象能不能继续、为什么不能继续、下一步会发生什么”，不是底层实现细节。
- 能减少用户记忆负担的地方就减少，不要让用户自己猜状态关系。

## 五、UI 设计约束

面向控制台、管理页、配置页、编辑器、运行面板这类工作界面：

- 先判断当前用户角色是谁，以及他此刻最想完成什么。
- 优先展示当前决策有用的信息：
  - 当前对象是否可继续操作
  - 当前正在做什么
  - 卡在哪一步
  - 用户现在能做什么操作
- 文案要短，但原因要明确。
- 不要把底层内部状态直接翻译成会误导用户的业务状态。
- 不要让按钮语义和后台真实状态冲突。
- 不要为了“全展示”牺牲扫描效率；状态要有主次。
- 保持现有页面风格一致，避免突然引入另一套状态语言或交互方式。
- 状态呈现与用户体验要平衡，不要为了技术完整性把界面做得啰嗦。

## 六、推荐工作顺序

对这类需求，推荐固定按这个顺序工作：

1. 先写真实场景下的端到端流程。
2. 再写状态归属表：
   - 状态名
   - owner
   - 更新触发方
   - 消费方
3. 再写合同模型：
   - 后端输出结构
   - 前端直接复用的字段
   - 哪些字段是领域字段，哪些字段是视图投影字段
   - 哪些值必须是枚举/布尔/数字，不能偷懒写字符串
4. 再写状态迁移或时序图。
5. 再写 UI 映射：
   - 内部状态
   - 当前进度文案
   - 展示状态文案
   - 按钮可操作性
6. 最后才进代码。

## 七、实现前自检清单

- 这个新增逻辑是不是放在了最合适的已有模块？
- 有没有绕过现有 owner，偷偷塞了第二套状态？
- 前后端是不是在描述同一件事时用了两套字段结构、两套状态名或两套枚举语义？
- 有没有本可结构化表达的值，被偷懒写成字符串后再到处 `if/else` 比较？
- 有没有把本该由下游系统、异步任务、后台服务或子进程上报的事实，提前在当前层本地猜了？
- 有没有把内部控制状态直接暴露成用户文案？
- 失败路径是不是只写了日志，没有落到账本和 UI？
- UI的错误日志有没有进行用户友好的方式优化？
- 这次改动是否真的降低了维护成本，而不是把复杂度挪到别处？

## 八、给助手的执行要求

当收到“设计一个功能 / 优化一个流程 / 修正状态流”这类请求时，应默认做到：

- 先用项目当前代码结构思考 owner 和边界。
- 先用真实用户流程思考前提条件和失败路径。
- 先用长期维护视角判断该放在哪一层。
- 先用当前页面使用者视角判断 UI 是否清楚、简洁、低负担。
- 先判断这次改动涉及的前后端合同模型应该挂在哪，是否已有可复用类型。
- 先判断哪些字段必须是结构化类型，哪些字段才应该是字符串。
- 如果发现需要新增状态、上下文或中间层，必须先回答：
  - 为什么现有 owner 不能承载？
  - 会不会形成第二套状态表达？
  - 对后续维护者是否更清晰？

## 九、前后端合同模型约束

这一条单独强调：

- 同一业务对象，只允许有一套主合同模型。
- 后端负责定义领域模型、账本模型、事件模型；前端负责消费这套模型，并在此基础上做视图投影。
- 前端可以新增 `ViewModel`、`DisplayModel`、`UIState`，但它们必须明确标注为“派生视图”，不能反过来成为业务真相源。
- 如果前端为了展示方便新增了一个状态名，必须回答它是否只是文案映射；如果不是，就应回到后端合同模型统一定义。
- 后端输出给前端的状态值，优先用可生成、可校验、可复用的枚举或联合类型，不要让前端自己维护一份字符串字典。
- 能从后端类型生成前端类型时，优先生成；不能生成时，也要保证字段名、可空性、枚举值、嵌套结构严格一致。
- 不允许出现这种情况：
  - 后端叫 `connection_status`，前端自己改成另一套并追加不同语义
  - 后端是枚举，前端落成任意字符串
  - 后端是结构化对象，前端拍平成零散字段后再手写拼装回去
  - 后端和前端分别维护一份相似但不完全相同的状态机

## 十、字符串使用约束

- 字符串不是默认类型，而是最后选择。
- 遇到以下语义，优先不用字符串：
  - 状态
  - 类型
  - 来源
  - 动作
  - 模式
  - 阶段
  - 开关
  - 优先级
- 优先顺序通常应是：
  - 枚举 / 联合类型
  - 布尔
  - 数字
  - 结构化对象
  - 最后才是字符串
- 如果不得不接收外部字符串值，应尽早在边界层解析、校验、转换，进入内部后使用结构化类型，不要把原始字符串一路传到业务层和 UI 层。
- 如果数据库历史上已经存了字符串状态，至少要在 domain 层提供明确枚举映射，不要在多处手写 `'running' / 'failed' / 'xxx'` 这类判断。

如果这些问题还没想清楚，不应直接开始写代码。
