# 视觉搜索与策略匹配执行备忘

这份备忘只记录当前已经确认的执行流职责划分，避免后续再把 `VisionSearch`、`handlePolicySet`、`handlePolicy` 混在一起。

## 1. 当前职责划分

### `VisionSearch`

- 作用：在 `handlePolicySet` 外部主动做一次视觉搜索。
- 目标：既支持 OCR 文本搜索，也支持 DET 标签筛选。
- 它不是策略匹配器，而是一个独立步骤。

当前字段约定：

- `det_res_var: Option<String>`
  - 默认“未绑定”。
  - 未绑定时，执行器回退到当前上下文最近一次 DET 结果。
- `ocr_res_var: Option<String>`
  - 默认“未绑定”。
  - 未绑定时，执行器回退到当前上下文最近一次 OCR 结果。
- `out_var: String`
  - 输出 `Vec<SearchHit>`。
- `out_det_var: Option<String>`
  - 默认“未绑定”。
  - 绑定后输出按规则筛出的 DET 结果。
- `out_ocr_var: Option<String>`
  - 默认“未绑定”。
  - 绑定后输出按规则筛出的 OCR 结果。

补充：

- `VisionSearch` 保留在 `handlePolicySet` 外部使用。
- `VisionSearch` 的定位是“先做一轮视觉搜索并产出结果变量”，不是“代替策略命中”。

### `handlePolicySet`

- 职责不变：处理策略集候选，并在候选链路里决定最终命中的策略。

记录链路：

- `while(任务条件) -> 截图 -> det -> ocr -> handlePolicySet`

### `handlePolicy`

- 职责不变：在 `handlePolicySet` 外部直接做策略匹配。

记录链路：

- `截图 -> det -> ocr -> handlePolicy`

补充：

- `handlePolicy` 的目的就是在 `handlePolicySet` 外部进行策略匹配。

## 2. 策略执行顺序

当前确认后的规则：

- `before_action` 改为在候选处理阶段执行。
- 也就是：
  1. 进入某个候选策略。
  2. 先执行 `before_action`。
  3. 再用最新上下文评估 `cond`。
  4. 命中后再执行 `after_action`。

这样改的原因：

- 不能强行限制 `before_action` 里的步骤类型。
- `before_action` 内部可能再次走 `截图 -> det -> ocr`。
- 如果先整体批量判断命中、后执行 `before_action`，就会错过这类“先动作再判断”的策略。

## 3. 点击文字输入类型

`点击文字` 当前应支持两种输入：

- `SearchHit[]`
- `OcrResult[]`

执行器处理原则：

- 如果输入变量能解析成 `SearchHit[]`，先从命中结果里提取对应 OCR 项再做点击目标选择。
- 否则按原来的 `OcrResult[]` 路径处理。

## 4. Rhai / Dynamic 当前放入范围

目前同意放入 Rhai `Dynamic` / scope 的结构化内容：

- `Vec<OcrResult>`
- `Vec<DetResult>`
- `Vec<SearchHit>`

另外，当前会进入脚本作用域的内容还包括：

- UI / 编辑器里配置的任务输入变量
  - 以 `input.*` 形式进入 scope。
  - 基础类型和 JSON 都能按变量读取。
- 部分运行时步骤产物
  - 例如 `runtime.ocrResults`
  - `runtime.detResults`
  - `runtime.searchHits`
  - 以及其它绑定到运行时变量的 JSON 结果

当前不在这次备忘里扩大的范围：

- 图像对象本身不作为普通结构化数据给 Rhai 做通用处理。
- 截图仍然是执行器内部能力，不作为“建议在代码块里直接处理的通用变量”。

补充：

- 之前那个把 `SearchHit` 压平成少数字段的 `search_hits_to_dynamic` 已经不再作为推荐方向。
- 当前更接近的目标是直接把完整结构化结果放进 `Dynamic`。

## 5. 为什么这样拆

最小认知模型如下：

- `det / ocr` 负责“产出视觉结果”
- `VisionSearch` 负责“在视觉结果上先做一轮外部搜索/筛选”
- `handlePolicySet` 负责“按策略集顺序驱动匹配与命中”
- `handlePolicy` 负责“单独做策略匹配”

这样四块职责不重叠，后续加“代码块”步骤时也更容易判断哪些东西该进 Rhai，哪些东西仍然留在执行器内部。
