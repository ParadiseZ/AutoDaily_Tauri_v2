# AutoDaily 项目交接文档

本文档用于给下一个 AI 快速接手当前项目，目标是回答 4 个问题：

1. 这个项目现在已经做到了什么。
2. 功能分别落在哪些页面、store、service 和 Tauri/Rust 命令上。
3. 数据存在哪里，依赖哪些外部环境。
4. 继续开发时最值得先看的位置和当前已知缺口。

## 1. 项目概览

- 项目类型：`Vue 3 + Vite + TypeScript` 前端，`Tauri 2 + Rust` 桌面端。
- 定位：多设备自动化工作台，核心围绕“设备管理、脚本管理、任务调度、运行日志、系统设置、账户/云端脚本”。
- 当前前端状态：
  - 主工作台 UI 已重写完成，路由和页面骨架完整。
  - 设备、脚本、任务、日志、设置、登录流程已经能走通主链路。
  - 脚本编辑器 `ScriptEditor.vue` 仍是占位工作区，未接真实图编辑/流程编辑能力。
- 当前后端状态：
  - SQLite、本地配置、日志系统、子进程管理、IPC、HTTP 客户端、脚本上传下载/克隆已具备基础实现。
  - 真正的“脚本执行引擎接业务任务图”仍未闭环，子进程调度部分有明确 TODO。

## 2. 目录与架构速览

### 2.1 顶层目录

- `src/`：Vue 前端。
- `src-tauri/`：Tauri 主应用入口。
- `src-tauri/crates/runtime_engine/`：当前大部分运行时基础设施实际实现都在这里，`src-tauri/src/infrastructure/*` 很多是 re-export。
- `src-tauri/crates/child_support/`：子进程运行时支持。
- `src-tauri/crates/runtime_common/`：ID、IPC、日志通用定义。
- `src-tauri/crates/vision_core/`：OCR / 视觉 / ONNX 相关能力。
- `tests/`：Playwright 测试目录，目前还是默认示例。
- `doc/`：现有项目文档目录。
- `server_backend/`：指向外部后端工程的符号链接，当前仓库内主逻辑并不直接调用这里的代码，但 Rust HTTP client 默认请求本地后端服务。

### 2.2 运行架构

- 前端通过 `@tauri-apps/api/core` 的 `invoke` 调用 Rust 命令。
- Rust 主进程负责：
  - 初始化 SQLite。
  - 初始化 Tauri store。
  - 初始化日志系统。
  - 启动 IPC server。
  - 初始化子进程管理器。
  - 响应前端页面发来的各类命令。
- 设备运行采用“主进程 + 每设备子进程”的模式：
  - 主进程下发 `Start / Pause / Stop / Shutdown / AddScript / RemoveScript` 等 IPC 命令。
  - 子进程把状态和日志通过 IPC 回传主进程。
  - 主进程再通过 Tauri event 向前端发 `device-status`、`device-error`、`child-log`。

## 3. 前端入口与公共链路

### 3.1 启动入口

- `src/main.ts`
  - 加载 `mockTauri`，方便浏览器本地调试。
  - 调用 `setupDebugBridge()`，把前端 console / unhandled error 转发给 Rust 侧 `frontend_debug_log_cmd`。
  - 注入 Pinia 和 Router。

- `src/App.vue`
  - 非 `/editor` 路由统一走 `MainLayout`。
  - 启动时做这些初始化：
    - `settingsStore.loadPreferences()`
    - `useThemeManager().initTheme()`
    - `userStore.hydrateAuthSession()`
    - `deviceStore.refreshAll()`
    - `userStore.checkProfile()`
    - `deviceStore.initIpcListeners()`
    - `logsStore.initListener()`

### 3.2 路由

`src/router/index.ts` 当前可用路由：

- `/tasks`：任务管理
- `/logs`：运行日志
- `/scripts`：本地脚本列表
- `/market`：脚本市场
- `/devices`：设备列表
- `/settings`：系统设置
- `/about`：关于
- `/editor`：脚本编辑器占位页

默认路由从 Tauri store 的 `routerStart` 读取，默认值是 `/tasks`。

### 3.3 状态管理

Pinia store 主要有：

- `device.ts`
  - 设备列表、在线设备、运行状态、CPU 核心数。
  - 监听 Tauri event：`device-status`、`device-error`。
  - 封装批量启停、进程启动/关闭。

- `script.ts`
  - 本地脚本列表、选中脚本、市场搜索结果、脚本任务缓存。
  - 负责创建空白脚本、上传、克隆、下载市场脚本。

- `task.ts`
  - 设备脚本队列、调度记录、时间模板。
  - 新增/删除 assignment 时，如果设备在线，会立即调用设备 IPC 命令同步队列。

- `settings.ts`
  - 界面偏好、本地系统偏好、日志配置、更新信息。
  - 本地偏好存在 Tauri store；运行时系统设置和日志设置会调用 Rust 命令。

- `user.ts`
  - 登录态、账户资料、认证弹窗状态。
  - 调用远端登录/注册/资料接口。

- `logs.ts`
  - 只维护前端显示用的子进程日志缓存。
  - 监听 Tauri event：`child-log`。

### 3.4 service 层

- `deviceService.ts`：设备 CRUD、运行状态、设备进程控制、子进程日志级别更新。
- `scriptService.ts`：本地脚本 CRUD、市场搜索、上传/下载/克隆、图片转 base64。
- `taskService.ts`：assignment、schedule、time template 查询与维护。
- `settingsService.ts`：日志配置、系统设置、检查更新。

### 3.5 invoke 包装

`src/utils/api.ts` 是统一调用入口：

- 对 `backend_*` 命令的 401/未登录场景做统一拦截。
- 登录失效时会自动打开 `AuthModal`。

## 7. 外部依赖与运行前提

### 7.1 本地后端服务

Rust `HttpClient` 的基址写死为：

- `http://localhost:8080/api`

所以只要用到这些能力，就需要本地后端服务启动：

- 登录 / 注册 / 找回密码

项目对 Android 设备自动化有明显依赖：

- 可配置 `adb.exe` 路径。
- 支持 TCP 直连、ADB server 按 IP / 按名称连接。

- `src-tauri/models/` 下已有内置模型。

## 10. 已知风险与注意事项

### 10.1 文档与代码状态不完全一致

- `README.md` 仍带有较旧的重构描述，和当前前端重写后的实际页面状态并不完全一致。
- `doc/child_process_work_record.md` 中提到的“前端尚未接入部分进程命令”已经部分过时，当前任务页和日志页已接入主要链路。

### 10.2 `save_script_tasks_cmd` 需要重点复查

从当前实现看，这个命令的 SQL `INSERT` 占位符数量与 `.bind()` 数量不一致，极可能在真正调用时出运行时错误。  
目前前端并没有接真实编辑器保存流程，所以这个问题还没暴露在主流程里，但下一个 AI 如果开始做编辑器，一定要先修这块。

### 10.3 脚本编辑器还没接入

- `/editor` 只是入口壳。
- 继续开发任务编辑、策略编辑前，建议先确认并固化 `script_tasks.data` 的内部结构。

### 10.4 远端服务地址写死

- 目前不是走环境变量，而是 Rust 常量。
- 如果要接测试环境/正式环境，优先改 `runtime_engine/src/infrastructure/http_client.rs`。

### 10.5 测试基本为空

- 现有 Playwright 测试对项目没有保护价值。
- 任何涉及编辑器、任务保存、设备控制的修改，最好补最少一层可执行验证。

## 12. 对下个 AI 最重要的结论

- 当前项目不是“从零开始”，而是“工作台主界面和主链路已成型，但脚本编辑器和执行引擎闭环还没完成”。

- 如果要做真正业务功能，必须同时看前端 store/service 和 Rust 命令实现，不然很容易只改到一半。
