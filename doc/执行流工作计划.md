## 一、当前先不做的事

- 不把 `DeviceQueue`、`FullScript / Task`、`Policy / PolicyGroup / PolicySet` 强行统一成同一套运行语义。
- 不实现 `account_id` 相关作用域。
- 不把 `process_api` 拆服务、`RuntimeContext` 继续拆层作为当前主计划项。

## 二、P1：补齐策略调试输入作用域

### 目标

- 让 `Policy / PolicyGroup / PolicySet` 调试运行自动装填 task 级 input 变量。
- 明确策略调试默认从当前屏幕截图开始，先完成 `DET / OCR` 观测，再进入命中判断。
- 不把 `time_template` 作用域选择作为当前主目标；没有模板信息时直接使用变量默认值。

### 需要完成的工作

1. 后端调整 `Policy / PolicyGroup / PolicySet` 调试链的输入作用域装填逻辑，不再让 `task=None` 直接过滤 task 级 input 变量。
2. 调试链在没有模板信息时，按脚本变量默认值装填，不再把时间模板 scope 作为必需前提。
3. 核对 `before_action / after_action` 的变量读取路径，保证策略命中前后步骤都能拿到需要的 task 级输入值。
4. 保持策略调试当前默认的截图 -> `DET / OCR` -> 命中判断链路，并把这条默认语义写入相关结构文档。
5. 清点 `load_debug_scope_queue_item()` / `build_debug_template_values_json()` 这类旧桥接逻辑，区分哪些还需要保留，哪些后续可以下沉或删除。

### 验收标准

- `Policy / PolicyGroup / PolicySet` 调试运行里的 before / after 步骤可以直接读取 task 级 input 变量。
- 没有 assignment / time template 信息时，策略调试仍能靠默认值正常运行。
- 策略调试仍保持“截图 -> `DET / OCR` -> 命中判断”的默认主链，不需要额外前置图像变量步骤。

## 三、P2：补齐 timeout detector 剩余缺口

### 目标

- 补完当前还没纳入 detector 的非动作路径。

### 需要完成的工作

1. 给条件求值路径补前进证据。
2. 给数据处理路径补前进证据。
3. 给首次观测、动作前准备、恢复装填等阶段补前进证据。
4. 整理 timeout 结果投影，区分：
   - 超时触发
   - 采取的动作
   - 最终结果

### 验收标准

- 长时间卡在纯计算或准备阶段时，也能被 detector 捕获。
- 前端不只看到一条 timeout 文本，还能看出触发后的动作结果。

## 四、P3：补齐前端运行态展示

### 目标

- 让当前投影视图不只停留在“文本提示”。

### 需要完成的工作

1. 扩展前端运行态 store 的结果结构。
2. 在 `TaskDevicePanel` 里补 timeout 行为结果展示。
3. 在日志或运行面板里补更稳定的调试结果回显。
4. 视情况补 action / trace / result 的结构化展示入口。

### 验收标准

- 可以从前端明确看出：
  - timeout 发生了没有
  - 采取了什么动作
  - 调试运行最后结果如何

## 五、P4：补测试

### 目标

- 让当前执行流主链的关键边界有可回归验证。

### 需要完成的工作

1. 补策略调试输入作用域装填测试。
2. 补 `Policy / PolicyGroup / PolicySet` 默认截图 -> `DET / OCR` -> 命中判断链测试。
3. 补 timeout 通知链与 `device-timeout` 投影测试。
4. 补 session reload 一致性测试。

### 验收标准

- 策略调试输入作用域、timeout、session reload 三条关键链路都有稳定回归测试。

## 六、低优先级记录

- `process_api` 服务化拆分。
- `RuntimeContext` 继续瘦身。
- 清理 `TimeoutAction` 的旧兼容动作名映射。

这些保留在结构文档里记录，但不作为当前主计划。
